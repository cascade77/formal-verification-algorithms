the whole idea revolved around three things: prove merge gives the right answer (the examples handle that), prove that merge of two sorted lists is sorted, and prove that mergesort itself always produces a sorted list. this file is about the first two. mergesort_sorted comes later.

---

`sorted` is an inductive proposition. this means it is not a function that returns true or false, it is a thing you build a proof of. there are three ways to prove a list is sorted, matching the three constructors. the empty list is sorted by definition, no condition needed. a single element list is always sorted. and a list with at least two elements is sorted when you can show the first element is `<=` the second, AND that the tail starting from the second element is also sorted. that last part is doing two things at once: a comparison between two adjacent elements, plus a recursive requirement on the rest. so `[1; 3; 2]` fails because even though `1 <= 3`, you also need `sorted [3; 2]`, which fails because `3 > 2`. the definition forces you to check the whole thing.

the naive way to write merge is to match on both lists at once. if `l1` is empty give back `l2`, if `l2` is empty give back `l1`, otherwise compare heads and recurse. rocq rejects this. the reason is that rocq needs to be convinced every recursive function terminates, and it does this by checking that some argument is getting strictly smaller on every call. in the branch where you take from `l1`, `l2` does not shrink. in the branch where you take from `l2`, `l1` does not shrink. rocq sees no single argument that consistently decreases and refuses the definition. the fix is to nest a helper function. the outer `merge` recurses on `l1`, so `l1` always shrinks on outer calls. the inner `merge_aux` recurses on `l2`, so `l2` always shrinks on inner calls. rocq checks each one independently and is satisfied. the `in merge_aux l2` at the end is just the actual call that runs the inner function; without it `merge_aux` would be defined but never executed and nothing would happen.

`split` takes a list and gives back two halves by alternating elements. `[1;2;3;4;5]` becomes `([1;3;5], [2;4])`. the `*` in the return type is a pair, not multiplication. the matching is on zero, one, or two-or-more elements, handling the edge cases cleanly.

`mergesort_aux` has an extra argument `n` called fuel. the reason is the same termination problem. if you write mergesort directly and recurse on the halves after splitting, rocq rejects it because the halves are not syntactic subterms of the original list. rocq does not know they are shorter, even though they obviously are. the workaround is to start with `n = length l` and pass `n-1` on each recursive call. rocq sees `n` shrinking and accepts it. the base case `n = 0` should never actually be hit on correct inputs but it gives rocq the guarantee it needs. `mergesort` is then just a clean wrapper that calls `mergesort_aux` with the right fuel.

now the lemmas. `merge_l_le` and `merge_r_lt` exist because `merge` uses a nested `let fix` and `simpl` does not always unfold it cleanly inside proof goals. so instead of fighting `simpl`, i wrote explicit rewrite rules: when `h1 <= h2` you can rewrite the merge to put `h1` first, and when `h1 > h2` you can rewrite it to put `h2` first. both proofs are just `simpl`, `rewrite`, `reflexivity`.

`sorted_cons_inv` and `sorted_cons_inv'` exist because rocq does not let you directly pull constructor arguments out of a hypothesis. if you have `H : sorted (x :: y :: l)`, you cannot just write `H.le` or whatever to get `x <= y`. you have to use `inversion`, which looks inside the constructor and creates fresh hypotheses for each piece. `sorted_cons_inv` does that and gives you both the comparison and the sorted tail as separate things. `sorted_cons_inv'` is the simpler version that just strips the head and gives you that the tail is sorted.

`le_of_leb` is a small annoyance. `Nat.leb a b = true` is a Bool computation and `a <= b` is a Prop. they mean the same thing but rocq treats them as completely different and will not convert between them automatically. this lemma bridges the gap using `Compare_dec.leb_complete` from the standard library.

`merge_sorted_cons` is a helper specifically written to make `merge_sorted` cleaner. the idea is: if some `h` is `<=` the head of an already-sorted list, then prepending `h` gives a sorted list. the proof just cases on whether the list is empty or not. having this as a separate lemma means the main proof can call it and immediately reduce each case into two cleaner subgoals, rather than doing all that work inline.

`sorted_head_le` says: if a list is sorted and `x` is somewhere in the tail, then the head `h` is `<= x`. not just `<=` the element immediately after it, but `<=` anything anywhere further along. this is proved by induction on the tail. the interesting subcase is when the tail has at least two elements. there you have `h <= a` from the sorted condition, and by the induction hypothesis you get `a <= x`, and then you need `h <= x`. i tried using `Nat.le_trans` here and rocq said that name does not exist in this version. i also tried `le_trans` with `Require Import Arith` and that did not work cleanly either. `lia` just handles it, since once all the pieces are in context `lia` figures out the transitivity automatically without you needing to know what the relevant lemma is called.

`merge_In` says: if `x` is in `merge l1 l2`, then `x` came from `l1` or from `l2`. proved by double induction, outer on `l1` and inner on `l2`. in each branch you use `merge_l_le` or `merge_r_lt` to rewrite the goal into something `simpl` can reduce, then case split on whether `x` is the head or is somewhere in the tail, and apply the induction hypothesis for the tail case. this lemma is used in `merge_sorted` to figure out where the head of a merged result came from, which is what lets you bound it from below.

`merge_sorted` is the main theorem. the outer structure is induction on `l1`, inner induction on `l2`. this gives four top-level cases. `l1 = []` is trivial: `merge [] l2 = l2` and you already know `l2` is sorted. `l2 = []` is symmetric. the interesting cases are when both lists are non-empty.

when `h1 <= h2`, you rewrite with `merge_l_le` so the goal becomes `sorted (h1 :: merge rest1 (h2 :: rest2))`. you apply `merge_sorted_cons` which splits it into two subgoals. the second subgoal, that `merge rest1 (h2 :: rest2)` is sorted, is exactly what `IH1` gives you after you strip `h1` from the sorted hypothesis on `l1`. the first subgoal is harder: you need `h1 <=` the head of that merge. the head is unknown, so you case on whether the merge is empty or not. if it is empty the goal becomes `h1 <= h1` which `lia` closes trivially. if it is non-empty the head is some `n`, and you assert that `n` is in the merge, then apply `merge_In` to learn it came from `rest1` or from `h2 :: rest2`. if it came from `rest1` then `sorted_head_le` gives you `h1 <= n` directly. if `n = h2` then `leb_complete` converts the boolean hypothesis to `h1 <= h2 = n`. if `n` is in `rest2` then `sorted_head_le` gives you `h2 <= n` and `lia` chains `h1 <= h2 <= n`.

when `h1 > h2` you do the symmetric thing with `merge_r_lt`. `h2 < h1` comes from converting the boolean with `Nat.leb_gt` and then `lia` handles the arithmetic.

a few bugs hit along the way. `Nat.le_trans` and `le_trans` both do not work cleanly in this version of rocq, `lia` is the fix. `Compare_dec.leb_gt` does not exist, `Nat.leb_gt` is the right name. bullet alignment after an `assert` block needs the closing `}` or rocq complains about an unfinished bullet. and several early attempts at the head comparison in `merge_sorted` tried to case on the structure of `rest1` directly and use `merge_l_le` to simplify, which only works when `rest1` is non-empty and its head is still `<= h2`. that falls apart in general. the correct approach is to not look at the structure of the lists at all, just use `merge_In` to trace where the head came from and `sorted_head_le` to bound it.

what is left: prove `mergesort_sorted`, which needs `merge_sorted` plus induction on `n` in `mergesort_aux` plus a lemma that `split` produces lists no longer than the original so the fuel is always valid. also prove that `mergesort` is a permutation, meaning every element appears the same number of times in the input and output. `count` is already defined for that.
